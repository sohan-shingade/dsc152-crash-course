Part A Learn: Core Concepts from Lab 9
01 The Big Picture — Model Selection vs Inference

Lab 9 uses the Lucas 2013 paper as a case study of what NOT to do. The paper studied whether casino poker rooms drive traffic to slot machines and table games — a "loss leader" argument.

THE central shortcoming: Lucas performed variable selection (backward/forward selection guided by p-values) and then used the resulting model for inference (hypothesis tests on coefficients). This is circular — you cannot use p-values to build a model and then trust the p-values from that model.

This is the model selection → inference problem from Lecture 13, and it's the single most important concept the professor is testing.

What Lucas did (WRONG)

  1. Looked at the data
  2. Used p-values to add/remove variables
  3. Added AR terms based on correlograms
  4. Added outlier dummies based on residuals
  5. Used the "final model" for inference

What should have been done (RIGHT)

  1. Pre-specify model from scientific reasoning
  2. Make summary tables and plots
  3. Run the primary analysis
  4. Check diagnostics (LINE conditions)
  5. Perform sensitivity analyses
Exam phrasing: "The author's workflow inappropriately mixes model selection with statistical inference." Memorize this sentence — it's the correct answer for Q1.2 and the foundation for many exam questions.
02 The Correct Workflow — Pre-Specified Model

The correct model for inference must be determined before looking at results. Here's the pre-specified model the lab says we should use:

$$\text{COIN\_pf} = \beta_0 + \beta_1 \cdot \text{RAKE} + \beta_2 \cdot \text{TUE} + \beta_3 \cdot \text{WED} + \beta_4 \cdot \text{THU} + \beta_5 \cdot \text{FRI} + \beta_6 \cdot \text{SAT} + \beta_7 \cdot \text{SUN}$$

Key features of this model:

  • Variables on raw dollar scale (not log-transformed)
  • Day of week included as a confounder (weekends are busier for everything)
  • Monday as the baseline level
  • NO variable selection — model is locked in before analysis
  • NO AR/MA terms, NO outlier dummy variables
Why does this differ from Lucas 2013?
(1) Lucas's final models were obtained via variable selection — objectively incorrect before inference.
(2) It would be hard to justify why each of the six models (3 resorts × 2 outcomes) should differ structurally if the scientific question is the same.

Categorical variables: all-or-nothing rule

Day of week is categorical with 7 levels. In any variable selection, you must keep ALL levels or remove ALL. You cannot remove individual days (like dropping Thursday alone because its p-value is not significant).

pre_specified_model.R
# Set Monday as reference level
casino$dow <- relevel(as.factor(casino$dow), ref = "Monday")

# Pre-specified model for Resort 1 slot profit
model1 <- lm(R1_COIN_pf ~ R1_RAKE + dow, data = casino)
summary(model1)

04 Diagnostic Checks — All 4 LINE Conditions

After running the primary analysis, you check the four conditions for valid linear regression inference. Here's the status for the casino model:

ConditionHow to CheckResult
LinearityResiduals vs fitted valuesLikely NOT violated
IndependenceResiduals vs time (day_num)MAYBE violated (cyclical patterns)
NormalityQQ-plot + Shapiro-Wilk testLIKELY VIOLATED
Equal varianceResiduals vs each predictorMAYBE violated (slight fan shape)
diagnostics.R
# Get residuals and fitted values
casino$residuals <- residuals(model1)
casino$fitted    <- fitted(model1)

# L: Linearity -- residuals vs fitted
ggplot(casino, aes(x = fitted, y = residuals)) + geom_point()

# I: Independence -- residuals vs time
ggplot(casino, aes(x = day_num, y = residuals)) + geom_point()

# N: Normality -- QQ plot + formal test
qqnorm(casino$residuals)
qqline(casino$residuals)
shapiro.test(casino$residuals)  # p < 0.05 => reject normality

# E: Equal variance -- residuals vs predictor
ggplot(casino, aes(x = R1_RAKE, y = residuals)) + geom_point()
Mnemonic: LINE — Linearity, Independence, Normality, Equal variance. You check ALL four, even if you suspect only one is problematic. On the exam, be specific about which plot you'd use for each.
05 Critical Exam Traps — 5 Things That Will Lose Points

Trap 1: log(mean(x)) != mean(log(x))

When building a summary table on the log scale, you must compute summary statistics after transforming, not before.

Wrong

log(summary_usd)
Takes log of already-computed means

Right

mean(log(casino$R1_COIN))
Computes mean of log-transformed values

Trap 2: "Using my data to justify a transformation"

Model selection (BAD)

"Looking at our data, the log scale gives a better linear fit, so we'll use log."

A priori (OK)

"Previous studies showed improved model fit on the log scale."

Using YOUR data to decide on a transformation IS model selection. Using prior studies is a priori justification.

Trap 3: Double-log coefficient misinterpretation

In a double-log model $\log(Y) = \beta_0 + \beta_1 \log(X)$, the coefficient $\beta_1 = 0.5$ means: "a 1% increase in $X$ is associated with a 0.5% increase in $Y$." NOT a 50% increase!

Similarly, $\beta_1 = 0.0646$ means a 1% increase in RAKE is associated with a 0.0646% increase in COIN — NOT 6.46%.

Trap 4: Constants on the log scale

If the model's constant/intercept is 13.8990 on the log scale, the actual dollar value is $e^{13.8990}$, not $13.8990.

Trap 5: "The data looks like an AR process, so add AR terms"

Visual similarity does NOT justify including AR terms in an inference model. Using correlograms from your data to decide whether to add AR(1) terms IS model selection — just as invalid as using p-values to select variables.

07 Exam Q Type: "Should you remove this non-significant variable?"

Expect a question where a colleague suggests removing a variable because its p-value is large.

Your colleague looks at the regression output and says: "The Thursday coefficient has $p = 0.82$, which is not significant. Let's remove it from the model to simplify things." What do you tell them?

Loses points

"We should keep it because it might matter."

Doesn't explain the model selection issue or the categorical variable rule.

Full marks

"Non-significant p-values do not justify removing pre-specified covariates. Removing them based on the data is model selection, which invalidates subsequent inference. Furthermore, day of week is categorical with 7 levels — you should not remove individual days. Either keep all levels or remove all."
Two-part answer pattern:
1. General rule: "Removing non-significant variables from a pre-specified model IS model selection, which invalidates inference."
2. Categorical trap: "Day of week is categorical — you keep ALL levels or remove ALL. Cannot remove Thursday alone."

Why you keep ALL day-of-week levels:

Day of week is a single categorical variable with 7 levels represented by 6 dummy variables (Monday = baseline). Removing just one dummy (Thursday) while keeping the rest is like removing one category from a variable — it doesn't make sense scientifically. You either control for day-of-week effects or you don't.
08 Exam Q Type: "Is this a valid justification for log-transforming?"

The exam will give you several reasons for log-transforming and ask which are valid. The key distinction: a priori (before seeing data) vs data-driven (model selection).

JustificationValid?Why
"Previous studies showed improved model fit on the log scale"VALIDA priori justification from external evidence
"Looking at our data shows the log scale gives a better linear fit"INVALIDUsing YOUR data to decide = model selection
"Log scale makes coefficients interpretable as elasticities"INVALIDConvenience of interpretation is not a statistical justification
The subtle distinction (Q1.3 vs Q4.2):
• Q1.3 says the primary reason for transformation should be to improve fit, not interpretation — this is about the goal of transforming.
• Q4.2 says if you use your own data to determine that log improves fit, that's model selection — this is about the evidence for transforming.

The resolution: you should transform to improve fit, but the evidence that fit improves should come from prior studies, not your current dataset.
Your Turn

A colleague says: "I plotted the residuals from the raw-scale model and saw non-constant variance. After log-transforming, the residuals look much better. So I'll use the log model for my inference." Is this appropriate?

No, this is not appropriate. The colleague is using their own data to decide on the transformation — looking at residuals from the raw model to justify switching to a log model. This is model selection, which invalidates the subsequent inference. A valid justification would be citing previous studies that found the log scale to be more appropriate for this type of data.

10 Bonus — AR(1) Simulation and When AR Terms Are Appropriate

Random walk simulation (phi_1 = 1)

The AR(1) formula: $y_t = c + \phi_1 \cdot y_{t-1} + \varepsilon_t$. With $c = 0$ and $\phi_1 = 1$, this is a random walk.

ar1_simulation.R
set.seed(1)
y <- numeric(217)
y[1] <- 14.2
for (i in 2:217) {
  y[i] <- y[i-1] + rnorm(1)  # c=0, phi_1=1, epsilon ~ N(0,1)
}
day <- 1:217
AR_df <- data.frame(day = day, y = y)
ggplot(data = AR_df, aes(x = day, y = y)) + geom_line()

Why the simulation looks different from actual data

Simulated AR(1) random walk

  • Drifts far from starting value (14.2)
  • No mean-reversion — once it drifts, it stays drifted
  • Cycles are NOT explained by day of week

Actual log(COIN) data

  • Stays within a relatively narrow range
  • Shows mean-reversion (comes back to average)
  • Cyclical patterns match weekday vs weekend effects
They do NOT look "remarkably similar." The simulated random walk drifts much further from 14.2 than the actual data. And even if they did look similar visually, that would NOT justify adding AR terms to an inference model.

When IS it appropriate to include AR terms?

ScenarioAppropriate?Why
Previous studies on similar data used AR(1) modelsYESA priori justification from external evidence
You looked at the correlogram of your residuals and saw autocorrelationNOUsing your data to decide = model selection
Your time series "looks like" it has AR behaviorNOVisual similarity is not a valid justification
The domain knowledge dictates temporal dependence (e.g., stock prices)YESScientific rationale specified before analysis

• Arima(y, xreg = X, order = c(1,0,0)) from the forecast package (note capital A)
